How connections get distributed
	Two valid models:
	Model 1: Shared Listening Socket
		Master creates one listening socket
		All workers inherit it
		All workers call accept()
		Kernel distributes incoming connections
		Result:
		Worker 1 gets some connections
			Worker 2 gets others
			No duplication

	Model 2: SO_REUSEPORT (cleaner design)
		Each worker creates its own listening socket
		All bind to same port
		Kernel load balances at connection time
		Result:
			Better scaling
			Less contention

you register a file descriptor (FD) (epoll registration)
--------------------------------------------------------
But not just the FD—you’re really registering:
FD + the events you care about + (optionally) some user data/context
Conceptually:
	register:
	    fd = 42
	    events = READ | WRITE
	    context = connection_state_object    
What epoll stores internally
	Epoll keeps something like:
		FD → interest (READ/WRITE/etc.)
		This is called the interest list.
What happens later
	When you call epoll_wait():
		It returns FDs that are ready
		Along with which event occurred
		fd=42 → READ ready
		fd=57 → WRITE ready

If you WANT true even distribution (using model 1 : workers race for new connections)
-------------------------------------------------------------------------------------
	Option A: SO_REUSEPORT (Model 2 idea)
		Each worker has its own listening socket
		Kernel distributes connections more evenly
	Option B: Accept mutex (Nginx-style tweak)
		Only ONE worker at a time is allowed to accept
		workers rotate responsibility
	Option C: Central dispatcher (not used in Nginx)
		One process accepts
		distributes to workers manually
